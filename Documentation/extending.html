Modifying the Code

Before You Start

Before making changes, take a few minutes to understand the overall structure. LootNScoot is organized around a central LNS table that acts as shared state across the entire application. Think of LNS as the "brain" of the script — every module reads from it and writes to it, and it is how the different parts of the code communicate with each other. There is no class hierarchy or object system; it is just one big Lua table passed to every module.

Here is how to orient yourself:

FilePurposeKey Contents
init.lua Main entry point, core logic The LNS table, command handler, rule functions, loot engine, event handlers, main loop
modules/settings.lua Settings definitions Settings (defaults), SettingsEnum (CLI mapping), Tooltips, SettingsNoDraw, TempSettings
modules/ui.lua All ImGui rendering RenderUIs(), RenderMainUI(), renderNewItem(), RenderModifyItemWindow(), etc.
modules/actor.lua Inter-process messaging RegisterActors(), Send(), the callback function with all action handlers
modules/db.lua SQLite database layer SetupItemsTable(), PrepareStatements(), CRUD functions
modules/performance.lua Frame timing / ImPlot Performance monitor rendering
modules/loot_hist.lua Console / loot history Scrollable log, theme system, report exports

Every module receives the central LNS table through a SetLNS() call during initialization. This is how modules access shared state without circular require() dependencies.

The best way to understand a code path is to enable debug mode: 

/lns debug on

This activates verbose logging to the console. You will see detailed tables printed for every actor message, rule decision, and setting change. Combined with the Debug Mailbox viewer, this gives you full visibility into what the script is doing.

Common Modifications

Adding a New Setting

Settings control LootNScoot's behavior. Adding a new one requires touching four locations in modules/settings.lua and then referencing it in your logic. The settings system is designed so that new settings are automatically picked up by the UI and CLI — you do not need to write any rendering or command-parsing code.

Step 1: Open modules/settings.lua and find the LNS_SETTINGS.Settings table (near the top of the file). Add your default value anywhere inside it:

LNS_SETTINGS.Settings = {
    ...
    MyNewSetting = false,    -- your new boolean setting
    MyNewNumber  = 50,       -- or a numeric setting
    ...
}

Step 2: In the same file, find the LNS_SETTINGS.SettingsEnum table. Add the CLI mapping. The key is the lowercase version users type in commands; the value is the ProperCase key in Settings. This mapping exists because Lua is case-sensitive, but users should not have to remember exact casing when typing commands.

LNS_SETTINGS.SettingsEnum = {
    ...
    mynewsetting = 'MyNewSetting',
    mynewnumber  = 'MyNewNumber',
    ...
}

This allows users to run /lns set mynewsetting on or /lns set mynewnumber 75.

Step 3: In the same file, find the LNS_SETTINGS.Tooltips table. Add a tooltip — this text appears when hovering over the setting in the UI.

LNS_SETTINGS.Tooltips = {
    ...
    MyNewSetting = "Description of what this setting does.",
    MyNewNumber  = "Description of this numeric setting.",
    ...
}

Step 4 (optional): If the setting should not appear in the Settings UI (because it is internal or managed by a dedicated UI), add it to SettingsNoDraw in the same file.

LNS_SETTINGS.SettingsNoDraw = {
    ...
    MyNewSetting = true,   -- hide from the UI
    ...
}

Step 5: Reference the setting in your logic (typically in init.lua).

if settings.Settings.MyNewSetting then
    -- do something when enabled
end

if settings.Settings.MyNewNumber > 25 then
    -- do something when the number exceeds 25
end

That is it. The Settings UI automatically picks up new settings because it iterates settings.Settings dynamically. Boolean settings get toggle switches; non-boolean settings get input fields. The command handler already supports any setting listed in SettingsEnum. Settings synchronization via actors also works automatically because the entire Settings table is broadcast.

Adding a New Loot Action

Warning: Adding a new loot action touches multiple files and affects the rule system, looting engine, and UI. Test thoroughly — a typo in the action name will cause items to be silently skipped. Make sure your action string is consistent (exact same casing) across all five locations listed below.

Loot actions (also called rules) determine what happens to an item. The built-in actions are: Ask, CanUse, Keep, Bank, Sell, Ignore, Destroy, Quest, and Tribute. Here is how to add a new one.

Step 1: Open init.lua and search for local validActions. Add your action. The key is lowercase (for CLI input); the value is ProperCase (the canonical form stored in rules).

local validActions = {
    ask     = "Ask",
    canuse  = "CanUse",
    keep    = 'Keep',
    bank    = 'Bank',
    sell    = 'Sell',
    ignore  = 'Ignore',
    destroy = 'Destroy',
    quest   = 'Quest',
    tribute = 'Tribute',
    myaction = 'MyAction',   -- your new action
}

Step 2: In the same file, search for local shouldLootActions (usually a few lines below validActions). Set to true if the item should be picked up from the corpse, or false if it should be left on the corpse.

local shouldLootActions = {
    Ask = false, CanUse = false, Keep = true, Bank = true,
    Sell = true, Destroy = false, Ignore = false,
    Tribute = false, Quest = false,
    MyAction = true,   -- true = pick up the item
}

Step 3: Add handling in the rule evaluation logic. The getRule() function and related decision functions in init.lua determine what rule applies to an item. You may need to add conditions that return your new action.

Step 4: Add handling in the item processing functions. If your action needs special behavior after looting (like Sell goes to a vendor, Tribute goes to a tribute master), add the logic in the appropriate processing function in init.lua.

Step 5: Open modules/ui.lua and search for local settingList. Add your action so it appears as a radio button in the New Item popup and Modify Item window.

local settingList = {
    "Ask", "CanUse", "Keep", "Ignore", "Destroy",
    "Quest", "Sell", "Tribute", "Bank",
    "MyAction",    -- your new action
}

Adding a New CLI Command

CLI commands are handled by LNS.commandHandler() in init.lua. The function receives variadic arguments from the /lns bind (mq.bind() is the MQ function that registers slash commands). Adding a new command means adding another if/elseif branch.

Step 1: Open init.lua and search for function LNS.commandHandler. It is a large function with many if/elseif branches — one per command.

Step 2: Add your handler following the existing pattern.

function LNS.commandHandler(...)
    local args = { ..., }
    ...

    if args[1] == 'mycommand' then
        local param = args[2] or 'default'
        -- your logic here
        Logger.Info(LNS.guiLoot.console,
            "MyCommand executed with param: \ay%s\ax", param)
        return
    end

    ...
end

Users can then run /lns mycommand somevalue in-game. Arguments are split by spaces and available as args[1], args[2], etc.

The bind itself is registered in LNS.setupBinds() and already routes all /lns commands to commandHandler(), so you do not need to register anything new.

Adding a New Actor Action

The Actor system is MQ's inter-process messaging system that lets LootNScoot instances exchange data. To add a new type of message:

Step 1: Open modules/actor.lua and find the callback function. It is a large if/elseif chain on the action field. Add your new action handler:

if action == 'my_new_action' and who ~= settings.MyName then
    -- handle the received message
    local someData = lootMessage.myField or 'default'
    -- update local state
    LNS.SomeTable[someData] = true
    settings.TempSettings.NeedSave = true
    Logger.Debug(guiLoot.console, dbgTbl)
    return
end

Step 2: Add the Send() call wherever the action originates. This could be in init.lua, ui.lua, or another module.

actors.Send({
    who     = settings.MyName,
    action  = 'my_new_action',
    myField = someValue,
    Server  = settings.EqServer,
})

Step 3: Define the message fields. There is no formal schema — you simply add whatever fields you need to the message table. The receiver accesses them from lootMessage. By convention, always include who, action, and Server.

Adding a New Database Table

The database layer uses lsqlite3 (a Lua binding for the SQLite database engine, loaded via PackageMan). All database operations live in modules/db.lua.

Step 1: Add a CREATE TABLE IF NOT EXISTS statement in SetupItemsTable() (or create a similar setup function).

function LNS_DB.SetupMyTable()
    LNS_DB.db:exec([[
        CREATE TABLE IF NOT EXISTS MyTable (
            id INTEGER PRIMARY KEY,
            name TEXT NOT NULL,
            value INTEGER DEFAULT 0,
            created_at TEXT DEFAULT CURRENT_TIMESTAMP
        )
    ]])
end

Step 2: Add prepared statements in PrepareStatements(). Prepared statements are pre-compiled SQL queries that are faster and safer than string concatenation.

function LNS_DB.PrepareStatements()
    ...
    LNS_DB.stmtInsertMyTable = LNS_DB.db:prepare(
        "INSERT OR REPLACE INTO MyTable (id, name, value) VALUES (?, ?, ?)")
    LNS_DB.stmtSelectMyTable = LNS_DB.db:prepare(
        "SELECT * FROM MyTable WHERE name = ?")
    ...
end

Step 3: Add Lua functions for CRUD (Create, Read, Update, Delete) operations that use the prepared statements.

function LNS_DB.InsertMyRecord(id, name, value)
    LNS_DB.stmtInsertMyTable:bind_values(id, name, value)
    LNS_DB.stmtInsertMyTable:step()
    LNS_DB.stmtInsertMyTable:reset()
end

function LNS_DB.GetMyRecord(name)
    LNS_DB.stmtSelectMyTable:bind_values(name)
    local row = LNS_DB.stmtSelectMyTable:nrows()
    LNS_DB.stmtSelectMyTable:reset()
    return row
end

Step 4: If you need fast lookups, add a Lua-side cache table on the LNS object and populate it on load.

LNS.MyCache = {}
-- Populate on init:
for row in LNS_DB.db:nrows("SELECT * FROM MyTable") do
    LNS.MyCache[row.id] = row
end

Adding a New UI Tab or Window

All UI rendering lives in modules/ui.lua and uses ImGui (Immediate Mode GUI) — a UI library where you redraw the entire interface every frame by calling functions like ImGui.Text() and ImGui.Button(). There is no persistent widget tree; if you stop calling a function, that widget disappears.

Step 1: Add a render function in modules/ui.lua. The ImGui.Begin() / ImGui.End() pair creates a window. The ##LNS suffix ensures a unique ImGui ID even if another script uses the same window title.

function LNS_UI.renderMyWindow()
    if not settings.TempSettings.ShowMyWindow then return end
    local open, show = ImGui.Begin("My Window##LNS", true)
    if not open then
        settings.TempSettings.ShowMyWindow = false
    end
    if show then
        ImGui.Text("Hello from my custom window!")
        if ImGui.Button("Close") then
            settings.TempSettings.ShowMyWindow = false
        end
    end
    ImGui.End()
end

Step 2: In the same file, find function LNS_UI.RenderUIs(). Add your call alongside the other window render calls, between the theme push and pop:

function LNS_UI.RenderUIs()
    local colCount, styCount = LNS.guiLoot.DrawTheme()
    ...
    LNS_UI.renderMyWindow()       -- add your window here
    ...
    if colCount > 0 then ImGui.PopStyleColor(colCount) end
    if styCount > 0 then ImGui.PopStyleVar(styCount) end
end

Step 3: Use a TempSettings flag for show/hide state (TempSettings is runtime-only, never saved to disk — perfect for UI toggle state). Add a button or CLI command somewhere that sets settings.TempSettings.ShowMyWindow = true.

To add a new tab instead of a new window, find the ImGui.BeginTabBar() call in drawItemsTables() and add an ImGui.BeginTabItem() block: 

if ImGui.BeginTabItem("My Tab") then
    -- render tab contents
    ImGui.Text("Tab content here")
    ImGui.EndTabItem()
end

The Event System

MacroQuest (MQ) can detect specific text patterns in EverQuest's chat output and trigger Lua callbacks. LootNScoot uses this for detecting loot-related game events. Events are registered in setupEvents() inside init.lua: 

function LNS.setupEvents()
    mq.event("CantLoot",
        "#*#may not loot this corpse#*#",
        LNS.eventCantLoot)
    mq.event("NoSlot",
        "#*#There are no open slots for the held item in your inventory#*#",
        LNS.eventNoSlot)
    mq.event("Sell",
        "#*#You receive#*# for the #1#(s)#*#",
        LNS.eventSell)
    mq.event("Forage",
        "You have scrounged up #*#",
        LNS.eventForage)
    mq.event("Novalue",
        "#*#give you absolutely nothing for the #1#.#*#",
        LNS.eventNovalue)
    mq.event("Tribute",
        "#*#We graciously accept your #1# as tribute, thank you!#*#",
        LNS.eventTribute)
end

The pattern syntax uses #*# as a wildcard (matches any text) and #1#, #2#, etc. as capture groups (the matched text is passed to the handler function as arguments).

To add a new event: 

mq.event("MyEvent",
    "#*#some game text with #1# capture#*#",
    function(line, captured1)
        Logger.Info(LNS.guiLoot.console,
            "MyEvent fired! Captured: %s", captured1)
        -- your logic here
    end)

The first argument to the handler is always the full matched line. Subsequent arguments are the captured groups (#1#, #2#, etc.).

Events are processed when mq.doevents() is called in the main loop. They do not fire in real time — they queue up and are processed on the next main loop iteration.

Debugging Tips

Since there is no automated test suite, all debugging happens live in the EverQuest client. Here are the tools available to you:

ToolHow to EnableWhat It Shows
Debug Mode /lns debug on Verbose logging of all rule decisions, actor messages, and state changes. Debug-level messages are printed to the console as formatted tables with fields like Lookup, Event, Action, ItemID, etc.
Mailbox Viewer /lns mailbox Real-time overlay showing every actor message with timestamp, subject (action), and sender. Requires debug mode to be on.
Performance Monitor Toggle via toolbar button ImPlot graph of frame timing. Useful for identifying UI render bottlenecks or slow database queries.
Console Search Filter box in console window Search through loot history and log messages by item name, date, zone, or action.

The Logger module supports multiple severity levels: 

Logger.Debug(LNS.guiLoot.console, "detailed trace info: %s", value)
Logger.Info(LNS.guiLoot.console, "normal operation: %s", value)
Logger.Warn(LNS.guiLoot.console, "something unexpected: %s", value)
Logger.Trace(LNS.guiLoot.console, "execution trace: %s", value)
Logger.Error(LNS.guiLoot.console, "something went wrong: %s", value)
Logger.Fatal(LNS.guiLoot.console, "critical failure: %s", value)
Logger.Help(LNS.guiLoot.console, "help text: %s", value)

The levels in ascending severity order are: debug (1), info (2), warn (3), trace (4), error (5), fatal (6), help (7). Each level takes the console reference as the first argument (so output goes to the console window) and supports string.format()-style formatting. You can also pass a table instead of a format string, and the Logger will print it as a structured key-value display.

Code Patterns to Follow

The SetLNS Pattern

Every module that needs access to the central LNS state table follows this pattern: 

local LNS

function MyModule.SetLNS(_LNS)
    LNS = _LNS
end

The main init.lua creates the LNS table and then calls SetLNS(LNS) on each module during initialization. This gives every module a reference to the same shared state without circular dependencies. If you create a new module, follow this same pattern and call YourModule.SetLNS(LNS) in init.lua's LNS.init() function.

TempSettings Flags

The UI runs in a render callback (called every frame by ImGui — the immediate-mode GUI library), while the main logic runs in a while loop with mq.delay() calls (mq.delay() pauses script execution for a duration). These two contexts cannot call each other directly. Instead, they communicate through flags in TempSettings — the runtime-only state table that is never saved to disk.

The pattern:

  1. The UI sets a flag: settings.TempSettings.SomeAction = true
  2. The main loop checks the flag: if settings.TempSettings.SomeAction then
  3. The main loop acts on it and clears it: settings.TempSettings.SomeAction = false

Examples from the codebase: NeedSave (trigger settings write), SendSettings (trigger actor broadcast), NeedsCleanup (trigger destroy pass), doBulkSet (trigger bulk rule application), ModifyItemRule (open the modify item window).

TempSettings is defined in modules/settings.lua with sensible defaults. Unlike Settings (which is persisted to disk), TempSettings is ephemeral — it resets every time the script restarts.

Error Handling

Lua does not have try/catch like other languages. Instead, use pcall() (protected call) to catch errors. pcall() runs a function and, if it throws an error, catches the error and returns it as a value instead of crashing the entire script. It returns two values: a boolean (true = success, false = error) and either the function's return value or the error message: 

local success, result = pcall(function()
    -- code that might error
end)
if not success then
    Logger.Error(LNS.guiLoot.console, "Operation failed: %s", result)
end

LootNScoot uses pcall() in two critical places:

  • Actor callbacks — the entire callback function is wrapped so a bad message never crashes the script.
  • Database operations — SQL errors are caught to prevent script termination.

When adding new code that interacts with external systems (database, actors, MQ TLOs), wrap it in pcall() to be safe.

Logger Usage

Always log through the Logger module rather than using print() or printf(). The Logger writes to the console window (visible in-game) and respects the theme system. (Note: if you need to run an in-game slash command from Lua, use mq.cmd('/commandname') — this executes a command as if you typed it in the EQ chat window. But for logging, always use the Logger.) 

Logger.Info(LNS.guiLoot.console, "message %s", arg)

The first argument is always the console reference (LNS.guiLoot.console). You can pass either a format string with arguments, or a Lua table which will be printed as structured key-value pairs. The table form is especially useful for debug output: 

Logger.Debug(LNS.guiLoot.console, {
    Lookup   = 'MyFunction()',
    Event    = 'Something happened',
    ItemID   = itemID,
    Rule     = currentRule,
})

EverQuest color codes work in Logger output: \ay = yellow, \ag = green, \ar = red, \at = teal, \ao = orange, \ax = reset to default.

Warning: Never call mq.delay() from inside an ImGui render function (anything called from RenderUIs()). ImGui render callbacks run on the UI thread every frame. A mq.delay() call will freeze the entire game client for the duration. All waiting, polling, and time-dependent logic must happen in the main loop (LNS.MainLoop()), not in render functions. Use TempSettings flags to communicate between the two.
Tip: When debugging rule decisions, enable debug mode (/lns debug on) and watch the console. Every rule evaluation logs the item name, the rule that was applied, and why. The structured table output from Logger.Debug() makes it easy to trace the decision chain for any specific item.
Note: LootNScoot has no automated test suite. All testing is done manually in-game. When you make changes, test them by running the script with /lua run lootnscoot and verifying behavior in the EverQuest client. Use debug mode liberally to confirm your changes are working as expected. Pay special attention to actor messages if your change affects multi-character features — test with at least two characters running simultaneously.
Tip: A practical debugging workflow: (1) enable debug mode with /lns debug on, (2) trigger the behavior you are testing (e.g., loot a corpse with /lns loot), (3) read the console output to see the exact code path that was taken. If you are debugging actor messages, also open the mailbox viewer with /lns mailbox to see messages in real time. If the script crashes, check the MQ console for the error message — Lua error messages include the file name and line number where the error occurred.
