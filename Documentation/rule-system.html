Rule Evaluation System

The rule system is the brain of LootNScoot. Every time an item is encountered on a corpse (a defeated enemy's remains in the game world), the script must decide what to do with it. This page explains the nine possible actions, how rules are looked up and prioritized, and the complete decision tree that getRule() walks through before returning a final verdict. Rules are stored in the SQLite database and cached in Lua tables for fast access.

The Nine Loot Actions

Every item rule maps to one of nine actions. Each action determines whether the item is picked up from the corpse and what happens to it afterward.

Action Picked Up? Description
Keep Yes Loot the item and keep it in your inventory permanently. Used for gear upgrades, consumables you want to use, or anything you intend to hold onto.
Sell Yes Loot the item. When you visit a merchant and issue /lns sellstuff, all Sell-flagged items are sold automatically. Subject to MinSellPrice and StackPlatValue thresholds.
Bank Yes Loot the item. When you visit a banker and issue /lns bankstuff, all Bank-flagged items are deposited. Tradeskill items can also be auto-banked if BankTradeskills is enabled.
Tribute Yes (when TributeKeep enabled) Loot the item if its tribute value meets the MinTributeValue threshold. When you visit a tribute master and issue /lns tributestuff, all Tribute-flagged items are tributed. Requires TributeKeep to be enabled for the item to actually be picked up during looting.
Quest|qty Yes (when LootQuest enabled, up to qty) Loot the item but only keep up to qty copies. The format is Quest|5 to keep 5, or just Quest to use the default QuestKeep setting (default 10). Once you have enough, the item is ignored. Requires LootQuest to be enabled. Counts are rechecked before each individual item pickup, not just during the initial corpse scan. This prevents over-looting when a corpse contains multiple copies of the same quest item (e.g., a corpse with 5 copies when you only need 1 more).
Destroy Yes (when DoDestroy enabled) The item is flagged for destruction. By default (DoDestroy disabled), items marked Destroy are skipped during looting. When DoDestroy is enabled, the item is picked up and destroyed. You can also issue /lns cleanup to destroy Destroy-flagged items already in your inventory. With AlwaysDestroy, Ignore-ruled items are also upgraded to Destroy on corpses.
CanUse No (evaluation only) The item is checked against your character's class and equipment slot compatibility. If your class can use the item, the decision is upgraded to Keep and the item is looted into inventory. If your class cannot, it becomes Ignore. This is primarily used for NoDrop items (items that cannot be traded once looted) where you only want to loot items your class is able to use. The script itself does not equip items — it places them into inventory via /autoinv. The EQ game client may auto-equip items into empty equipment slots if the character can wear them; if the slot is already occupied, items simply go to inventory.
Ask No The item is skipped during looting and added to the New Items list. You must manually decide what to do with it through the UI or CLI commands. This is the default for items with no existing rule when UseAutoRules is disabled.
Ignore No The item is skipped completely. The script will not pick it up and will not prompt you about it (unless AlwaysDestroy is enabled, in which case Ignore items are upgraded to Destroy).

The shouldLootActions Table

Internally, the script uses a lookup table called shouldLootActions to determine which actions result in actually picking up an item: 

local shouldLootActions = {
    Ask     = false,
    CanUse  = false,
    Keep    = true,
    Bank    = true,
    Sell    = true,
    Destroy = false,
    Ignore  = false,
    Tribute = false,
    Quest   = false,
}

The values shown above are the initial defaults. At runtime, three entries are dynamically updated based on settings:

  • shouldLootActions.Tribute is set to the value of TributeKeep
  • shouldLootActions.Quest is set to the value of LootQuest
  • shouldLootActions.Destroy is set to the value of DoDestroy

This means Tribute items are only picked up when TributeKeep is enabled, Quest items when LootQuest is enabled, and Destroy items when DoDestroy is enabled. The shouldLootActions table is used both for determining whether to pick up items and for tracking skipped items.

Rule Priority Chain

When getRule() needs to find a rule for an item, it searches through multiple sources in a strict priority order. The first match wins.

Rule Lookup Priority (lookupLootRule): +=======================================================+ | | | 1. Personal Rules (in-memory) | | LNS.PersonalItemsRules[itemID] | | -- Fastest: checks a Lua table directly | | -- If found, return immediately | | | | 2. Global Rules (in-memory) | | LNS.GlobalItemsRules[itemID] | | -- Same speed, checked second | | -- If found, return immediately | | | | 3. Personal Rules (database) | | db.CheckRulesDB(itemID, CHECK_DB_PERSONAL) | | -- SQL query against [CharName]_Rules table | | -- Slower but catches rules not yet cached | | | | 4. Global Rules (database) | | db.CheckRulesDB(itemID, CHECK_DB_GLOBAL) | | -- SQL query against Global_Rules table | | | | 5. WildCard Patterns | | LNS.CheckWildCards(itemName) | | -- Iterates all wildcard patterns | | -- Uses Lua pattern matching on item name | | -- If matched, creates a new rule entry | | | | 6. No match found: return 'NULL' | | (getRule then handles as a new item) | | | +=======================================================+

Step 1-2: In-Memory Lookup

The fastest path. lookupLootRule() first checks the Lua tables LNS.PersonalItemsRules and LNS.GlobalItemsRules, which are dictionaries keyed by item ID. If a rule was loaded from the database during a previous lookup, it gets cached here for instant access on subsequent checks. This caching is critical for performance: a single corpse may have a dozen items, and the looting engine processes multiple corpses per cycle. Without caching, each item would require a database query, which is far slower than a Lua table lookup. Personal rules are checked before Global, so a Personal rule always overrides a Global rule for the same item. This is by design — it allows per-character customization without affecting other characters sharing the same server database.

Step 3-4: Database Lookup

If the item is not in the in-memory cache, the script queries the SQLite database. It checks the Personal table ([CharName]_Rules) first, then the Global_Rules table. Any rule found in the database is immediately cached into the in-memory tables so subsequent lookups are instant.

Step 5: WildCard Patterns

If no explicit rule exists for this specific item ID, the script checks wildcard patterns. These are glob-style patterns that match against the item's name rather than its ID. If a wildcard matches, a new rule entry is created for the item using the wildcard's assigned action.

Step 6: No Match (New Item)

If none of the above sources produce a rule, lookupLootRule() returns 'NULL'. The calling function getRule() then treats this as a new, never-before-seen item and enters the auto-rules logic or defaults to Ask.

Personal rules always win. If you set a Personal rule for an item, it overrides any Global rule, any wildcard pattern, and any auto-rule logic. This is by design — it lets each character have item-specific overrides without affecting other characters on the same server.

The getRule() Decision Tree

After lookupLootRule() returns a rule (or 'NULL' for new items), getRule() applies a series of checks and overrides based on your settings and the item's properties. This is the complete decision tree:

getRule(item, fromFunction) Decision Tree: ============================================ item is nil or ID is 0? YES --> return 'NULL' Lookup rule via lookupLootRule() | v AlwaysEval enabled AND rule is Global? YES --> Clear non-critical rules (keep Quest/Keep/Destroy/ CanUse/Ask/Bank), force re-evaluation | v Rule == 'NULL' (new item)? YES --> Enter Auto-Rules Logic (see next section) | Creates rule and adds to NewItems list | v AlwaysAsk enabled OR rule == 'Ask'? YES --> Add to NewItems, return 'Ask' | (MasterLooting: also notify via actors) | v Rule == 'Ignore'? +--> DoDestroy AND AlwaysDestroy enabled? | YES --> return 'Destroy' (upgrades Ignore to Destroy) | NO --> return 'Ignore' | v Rule == 'Destroy'? YES --> return 'Destroy' | v StackableOnly enabled AND item not stackable? YES --> return 'Ignore' | v Item is Lore AND already owned? YES --> return 'Ignore' (add to skipped list) | v Item is NoDrop AND LootNoDrop disabled? YES --> return 'Ignore' (add to skipped list) | v Class check (Global rules, not Sell/Ask/Tribute/Quest): Your class not in item's class list? YES --> return 'Ignore' | v Rule == 'CanUse' AND item is equippable? Check classes, then check wearable slots: Class can use --> decision = 'Keep' (looted to inventory) Class cannot use --> decision = 'Ignore' | v Rule == 'Sell' or 'Tribute' (Global only)? Check sell/tribute thresholds: Below MinSellPrice --> may change to Ignore Below MinTributeValue --> may change to Ignore | v Bag space check: No free slots AND item not stackable with free stack? YES --> return 'Ignore' ("OUT OF BAG SPACE") | v LootAugments enabled AND item is Augmentation (Global)? YES --> decision = 'Keep' | v KeepSpells enabled AND item is Spell/Song (Global)? YES --> decision = 'Keep' | v Rule contains 'Quest'? Check LootQuest setting Check quantity (Quest|qty format): Have enough --> return 'Ignore' Need more --> check class restrictions Wrong class --> return 'Ignore' Right class --> decision = 'Quest' | v MasterLooting enabled? Rule is Ignore (not Personal) or Destroy --> return 'Ignore' Otherwise --> add to MasterLootList, return 'MasterLooter' | v Return final lootDecision

The tree is evaluated top-to-bottom. Once a return statement is hit, evaluation stops immediately. This means earlier checks (like AlwaysAsk and StackableOnly) act as "short circuits" that skip all later logic.

Auto-Rules Logic

When an item has no existing rule (lookupLootRule() returned 'NULL') and UseAutoRules is true (the default), the script attempts to assign a sensible default rule based on the item's properties. This logic lives inside getRule() in the "NEW RULES" section.

Auto-Rules Logic (UseAutoRules == true): ========================================= Is item NoDrop? YES --> Are LootNoDropNew AND LootNoDrop both enabled? | YES --> Is item equippable? | | YES --> rule = 'CanUse' | | NO --> return 'Ask' (don't add to DB) | NO --> return 'Ask' (don't add to DB) | NO --> Is item equippable? YES --> rule = 'Keep' NO --> Does item have a sell price > 0? YES --> rule = 'Sell' NO --> Does item have tribute value > 0? YES --> rule = 'Tribute' NO --> rule = 'Ask'

After determining the auto-rule, the script calls LNS.addNewItem(item, rule, link, corpseID, addToDB) to:

  • Add the item to the LNS.NewItems table for UI display
  • Increment LNS.NewItemsCount (triggers the gold coin indicator)
  • Optionally write the rule to the database (skipped for NoDrop items when LootNoDrop settings are disabled)

Even though auto-rules assign a rule automatically, the item still appears in the New Items list so you can review and override the decision. If AutoShowNewItem is enabled, the New Items window pops up automatically.

WildCard Patterns

WildCard rules let you create a single rule that applies to many items based on their name. They use standard Lua pattern syntax (not glob syntax). Common pattern elements include:

Pattern Element Meaning
.* Match any number of any characters
. Match exactly one character
%x Lua pattern escape (e.g., %. matches a literal dot)
^ / $ Anchor to start / end of string

The NormalizePattern() function lowercases literal characters in the pattern while preserving %-escaped sequences exactly as written. Item names are also lowercased before matching, so patterns are case-insensitive.

Examples

Pattern Matches Use Case
spell:.* Spell: Minor Healing, Spell: Gate, etc. Set a rule for all spell scrolls
.*rune.* Greater Rune, Rune of Karana, etc. Match any item with "Rune" in its name
song:.* Song: Hymn of Restoration, etc. Set a rule for all song scrolls
tattered .... page Tattered Blue Page, Tattered Gold Page Match four-letter color names with . single-character wildcards

Wildcards are stored in the database and loaded into LNS.WildCards (an array of {wildcard, rule} entries) at startup. You can add and remove wildcards through the UI or by setting TempSettings flags. When a wildcard matches a new item, a concrete rule entry is created for that specific item ID in the Global Rules table, so the wildcard only needs to match once per item.

Wildcards are only checked during lookupLootRule() when no Personal or Global rule exists for the item (step 5 in the priority chain). Once a match creates a concrete rule, subsequent lookups find that rule in steps 1-4 and never reach the wildcard check again.

Adding and Modifying Rules

There are several ways to create or change rules:

  • Through the UI — The Global Rules and Personal Rules tabs let you search for items, click the edit button, and set a rule. The New Items popup shows undecided items with dropdown selectors. See the User Interface page for details.
  • Through CLI commands — Commands like /lns set can modify settings that affect rule behavior. See CLI Commands.
  • Automatically — When AddNewSales is enabled, selling an item while the script is running automatically flags it as Sell. Similarly, AddNewTributes flags tributed items as Tribute.
  • Via actors — Other characters can share rules through the multi-character system.
  • Bulk operations — The UI supports selecting multiple items and setting them all to the same rule at once.
Be careful with AlwaysDestroy. When both DoDestroy and AlwaysDestroy are enabled, all items with an Ignore rule are upgraded to Destroy. This means items will be actively destroyed from corpses to clear them. Non-Quest items marked Ignore will be deleted. Make sure you have intentionally set Ignore rules before enabling this combination — there is no undo for destroyed items.
Personal overrides Global. If you want a specific character to handle an item differently from the global rule, create a Personal rule for that character. Personal rules are always checked first and completely override any Global rule, wildcard match, or auto-rule logic for that item on that character.

Special Items and Edge Cases

NEVER_SELL List

A small hardcoded table in init.lua prevents certain currency-like items from being sold: 

local NEVER_SELL = {
    ['Diamond Coin'] = true,
    ['Celestial Crest'] = true,
    ['Gold Coin'] = true,
    ['Taelosian Symbols'] = true,
    ['Planar Symbols'] = true,
}

These items are skipped during sell processing regardless of their rule.

Master Looter Redirect

When MasterLooting is enabled, the final step of getRule() redirects most items to the master loot system instead of looting them directly. Items with Ignore (non-Personal) or Destroy rules are still ignored, but everything else is added to LNS.MasterLootList and returns 'MasterLooter' as the action. The master looter then assigns items to specific characters. See Multi-Character for details.

Lore Items

Lore items are items you can only have one of — this is an EverQuest item flag enforced by the game client. If mq.TLO.FindItemCount(itemName) plus bank count is greater than zero, the item is automatically Ignored regardless of its rule. Lore checks happen early in the decision tree (before class checks, bag space checks, etc.) so they short-circuit quickly. This early placement is intentional: if the game would reject the pickup anyway, there is no point evaluating class restrictions or bag space.

The looting engine also separates lore and non-lore items into two processing passes. Non-lore items are looted first because a failed lore pickup (the game rejecting a duplicate) can disrupt the loot window state, and processing safe items first minimizes that risk.
Loot-time recheck: In addition to the scan-phase lore check described above, the looting engine rechecks FindItemCount + FindItemBankCount > 0 immediately before each individual item pickup. The game itself prevents picking up duplicate lore items, but attempting to do so can stall the loot cycle on that corpse. The recheck skips the attempt entirely, keeping the loot loop moving smoothly.

Spell and Song Scrolls

When KeepSpells is enabled (default true), any item whose name starts with "Spell:" or "Song:" is automatically kept, overriding Global rules (but not Personal rules). This ensures spell scrolls are collected even if a Global rule would otherwise sell or ignore them.
