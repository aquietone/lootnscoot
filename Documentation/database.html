Database Layer

Overview

LootNScoot stores all persistent data in three separate SQLite database files. SQLite is a lightweight, file-based relational database—no server process is needed. Each database file lives on disk and is accessed directly by the Lua script.

The SQLite library is loaded via require 'lsqlite3' in db.lua. The init.lua entry point uses PackageMan.Require('lsqlite3') to ensure the package is installed first (PackageMan is MQ's built-in package manager that automatically downloads and caches Lua libraries on first use). All three databases are opened with WAL (Write-Ahead Logging) journal mode: 

db:exec("PRAGMA journal_mode=WAL;")

WAL mode is a SQLite configuration that separates reads from writes. Instead of locking the entire database file during a write, changes are appended to a separate log file. This is critical for LootNScoot because multiple EverQuest characters (each running their own LootNScoot instance in a separate process) share the same database files on disk. Without WAL, one character writing a rule would block all others from reading. With WAL, readers and writers can operate simultaneously. This is what makes multi-character rule sharing work.

All three database files are stored under: 

[MQ Resources]/LootNScoot/[ServerName]/

The paths are constructed in modules/db.lua: 

LNS_DB.RulesDB   = resourceDir .. "/LootNScoot/" .. server .. "/AdvLootRules.db"
LNS_DB.ItemsDB   = resourceDir .. "/LootNScoot/" .. server .. "/Items.db"
LNS_DB.HistoryDB = resourceDir .. "/LootNScoot/" .. server .. "/LootHistory.db"

Items.db

The Items database stores detailed stat information about every item LootNScoot has encountered. It serves as a local item catalog so the script can evaluate items without querying the game client repeatedly.

Schema

The Items table has 60+ columns. Here are the key groups:

Column GroupColumnsDescription
Identity item_id (INTEGER PRIMARY KEY), name (TEXT NOT NULL) Unique game item ID and display name
Flags nodrop, notrade, tradeskill, quest, lore, augment, stackable, collectible Boolean flags (stored as INTEGER 0/1). Key flags: nodrop = item cannot be traded once looted; lore = only one copy can be held; tradeskill = used in crafting recipes
Value sell_value, tribute_value Vendor sell price (in copper) and tribute point value
Stacking stack_size Maximum stack size for stackable items
Core Stats strength, dexterity, agility, stamina, intelligence, wisdom, charisma Base character stat bonuses
Combat Stats hp, mana, ac, attack, damage, haste, regen_hp, regen_mana, spelldamage, spellshield, strikethrough, item_range Hit points, mana, armor class, and other combat values
Resists svfire, svcold, svdisease, svpoison, svcorruption, svmagic Resistance to various damage types
Heroic Stats heroicstr, heroicdex, heroicagi, heroicsta, heroicint, heroicwis, heroiccha, heroicsv* Heroic (enhanced) versions of base stats and resists
Restrictions classes (INTEGER bitmask), class_list (TEXT), races (INTEGER bitmask), race_list (TEXT) Which classes and races can use the item
Physical weight, item_size, weightreduction Item weight (tenths of a unit), size category, bag weight reduction
Miscellaneous clickable (TEXT), augtype, icon, link Click effect name, augment type mask, display icon ID, item link string

Two indexes are created for fast lookups: 

CREATE INDEX IF NOT EXISTS idx_item_name ON Items (name);
CREATE INDEX IF NOT EXISTS idx_item_id ON Items (item_id);

Adding Items

When a new item is encountered (on a corpse, in a forage, etc.), the looting engine calls LNS.addToItemDB(mq_item). This function extracts all stats from the MQ item TLO (Top-Level Object — MQ's read-only API for accessing game data, e.g., mq_item.NoDrop() returns whether the item is NoDrop) and populates the LNS.ALLITEMS table in memory, then calls db.AddItemToDB() to persist it.

AddItemToDB() uses the prepared statement ADD_ITEM_TO_DB, which is an INSERT with an ON CONFLICT(item_id) DO UPDATE clause. This means existing items are updated with the latest stats automatically.

The LNS.addMyInventoryToDB() function (triggered by /lns importinv) bulk-imports every item in your inventory and bank into Items.db.

Querying

LNS_DB.GetItemFromDB(itemName, itemID, query, useItemIDStmt) retrieves item data. When useItemIDStmt is true, it uses the prepared statement GET_ITEM_FROM_DB for fast ID-based lookup. Otherwise, it prepares a custom SQL query (for name-based or pattern searches).

LNS_DB.FindItemInDB(query, param, maxResults) is a more flexible search that returns a table of matching items, limited to maxResults entries.

AdvLootRules.db

The rules database stores all loot decisions. It contains four tables that define how items should be handled when encountered.

Global_Rules Table

The primary rules table, shared across all characters on the same server. 

CREATE TABLE IF NOT EXISTS Global_Rules (
    item_id INTEGER PRIMARY KEY NOT NULL UNIQUE,
    item_name TEXT NOT NULL,
    item_rule TEXT NOT NULL,
    item_rule_classes TEXT,
    item_link TEXT
);
ColumnTypeDescription
item_idINTEGER (PK)The unique game item ID. Negative IDs indicate items not yet found in Items.db.
item_nameTEXTThe item's display name.
item_ruleTEXTThe loot action: Keep, Sell, Bank, Ignore, Destroy, Quest, Quest|N, Tribute, Ask, CanUse.
item_rule_classesTEXTComma-separated class short names (e.g., "WAR,PAL,SHD") or "All".
item_linkTEXTThe MQ item link string (clickable in-game) or "NULL".

Personal Rules

Each character has their own personal rules table, named [CharName]_Rules. The schema is identical to Global_Rules: 

CREATE TABLE IF NOT EXISTS CharName_Rules (
    item_id INTEGER PRIMARY KEY NOT NULL UNIQUE,
    item_name TEXT NOT NULL,
    item_rule TEXT NOT NULL,
    item_rule_classes TEXT,
    item_link TEXT
);

Personal rules override global rules during rule evaluation. This design allows individual characters to have different loot preferences for the same item without modifying the shared Global rules. For example, a warrior might set a Personal "Keep" rule for plate armor that other classes would "Sell".

WildCards Table

CREATE TABLE IF NOT EXISTS WildCards (
    wildcard TEXT PRIMARY KEY NOT NULL UNIQUE,
    rule TEXT NOT NULL
);

Wildcard rules use Lua pattern matching (similar to regular expressions) to match item names. For example, a wildcard of "Spell: .*" could match all spell scrolls. Wildcards are loaded sorted by length (longest first) so more specific patterns take priority over general ones.

SafeZones Table

CREATE TABLE IF NOT EXISTS SafeZones (
    zone TEXT PRIMARY KEY NOT NULL UNIQUE
);

Stores zone short names where looting is completely disabled. When your character enters a safe zone, lootMobs() returns immediately without scanning for corpses.

Rule CRUD Operations

Rules are managed through several database functions:

FunctionOperationDescription
UpsertItemRule() INSERT ... ON CONFLICT DO UPDATE Creates a new rule or updates an existing one. Uses prepared statements INSERT_RULE_GLOBAL or INSERT_RULE_PERSONAL.
DeleteItemRule() DELETE FROM ... WHERE item_id = ? Removes a rule entirely from the specified table.
BulkSet() Batch INSERT/DELETE in a transaction Applies a rule change to many items at once. Wraps all operations in BEGIN/COMMIT. Sends actor message to notify other characters.
CheckRulesDB() SELECT Checks if a rule exists for a given item ID. Returns found status, rule, classes, and link.
AddWildCard() / DeleteWildCard() INSERT OR REPLACE / DELETE Manages wildcard pattern rules. Sends actor messages to sync across characters.
AddSafeZone() / RemoveSafeZone() INSERT OR IGNORE / DELETE Manages the safe zone list.

Importing

LNS_DB.ImportOldRulesDB(path) imports rules from a different AdvLootRules.db file (e.g., from a backup or another server). It reads both Global_Rules and Normal_Rules from the source, resolves item IDs against the local Items.db where possible, and inserts them into the current Global_Rules table using upsert.

LNS_DB.MigrateNormalToGlobal() is a one-time migration that moves any entries from a legacy Normal_Rules table into Global_Rules. Global rules win on ID conflicts. After migration, Normal_Rules is dropped.

LootHistory.db

The history database records every looting action taken, providing an audit trail and enabling the loot report feature.

Schema

CREATE TABLE IF NOT EXISTS LootHistory (
    "id" INTEGER PRIMARY KEY AUTOINCREMENT,
    "Item" TEXT NOT NULL,
    "CorpseName" TEXT NOT NULL,
    "Action" TEXT NOT NULL,
    "Date" TEXT NOT NULL,
    "TimeStamp" TEXT NOT NULL,
    "Link" TEXT NOT NULL,
    "Looter" TEXT NOT NULL,
    "Zone" TEXT NOT NULL
);
ColumnTypeDescription
idINTEGER (auto)Auto-incrementing row ID
ItemTEXTItem name
CorpseNameTEXTName of the corpse it came from
ActionTEXTWhat was done: "Global Keep", "Personal Sell", "Left", "Ask", "Destroyed", etc.
DateTEXTDate string in YYYY-MM-DD format

Querying

FunctionQueryReturns
LoadHistoricalData() SELECT DISTINCT Date FROM LootHistory ORDER BY Date DESC List of all dates that have loot records
LoadDateHistory(date) SELECT * FROM LootHistory WHERE Date = ? All loot records for a specific date
LoadItemHistory(name) SELECT * FROM LootHistory WHERE Item LIKE ? All loot records matching an item name (supports partial matches)

Deduplication

LNS_DB.CheckHistory(itemName, corpseName, action, date, timestamp) prevents duplicate entries by checking if the same item/corpse/action combination was recorded within the last 60 seconds. It compares timestamps by converting HH:MM:SS to total seconds. If the time difference is 60 seconds or less, the insert is skipped.

Prepared Statements

LNS_DB.PrepareStatements() pre-compiles frequently used SQL queries into prepared statements at startup. Prepared statements are faster than compiling SQL on every execution because the database engine parses and optimizes the query once, then reuses the execution plan. Parameters are bound with :bind_values().

If you add new database queries when modifying the code, follow the existing pattern: define the SQL string, prepare it in PrepareStatements(), and use :bind_values() with :step() / :reset() at call sites. Never concatenate user input into SQL strings directly — always use parameter binding.
Statement NameDatabasePurpose
ADD_ITEM_TO_DBItems.dbUpsert an item with all 61 stat columns

CHECK_HISTORYLootHistory.dbCheck for recent duplicate history entry
INSERT_HISTORYLootHistory.dbInsert a new loot history record
INSERT_RULE_GLOBALAdvLootRules.dbUpsert a rule into Global_Rules
INSERT_RULE_PERSONALAdvLootRules.dbUpsert a rule into [CharName]_Rules
CHECK_DB_GLOBALAdvLootRules.dbLook up a rule from Global_Rules by item ID
CHECK_DB_PERSONALAdvLootRules.dbLook up a rule from personal rules by item ID
GET_ITEM_LINK_GLOBALAdvLootRules.dbGet the item link from Global_Rules

UPDATE_RULE_LINK_GLOBALAdvLootRules.dbUpdate an item link in Global_Rules

Lua-Side Caching

While the SQLite databases are the source of truth, LootNScoot maintains several Lua tables in memory for fast access during the looting loop (where speed matters most). This avoids issuing a SQL query for every item on every corpse — a Lua table lookup by key is effectively instant, while a database query involves disk I/O and SQL parsing overhead. The rule priority chain checks these caches first (steps 1-2) and only falls back to the database (steps 3-4) for items not yet cached:

Cache TableSourceContents
LNS.GlobalItemsRulesGlobal_RulesMap of item_id to rule string
LNS.GlobalItemsClassesGlobal_RulesMap of item_id to classes string
LNS.PersonalItemsRules[CharName]_RulesMap of item_id to rule string
LNS.PersonalItemsClasses[CharName]_RulesMap of item_id to classes string
LNS.ItemLinksBoth rule tablesMap of item_id to item link string
LNS.ItemNamesBoth rule tables + Items.dbMap of item_id to item name
LNS.ItemIconsItems.dbMap of item_id to icon ID
LNS.ALLITEMSItems.dbFull item data objects keyed by item_id
LNS.WildCardsWildCards tableOrdered list of {wildcard, rule} pairs
LNS.SafeZonesSafeZones tableSet of zone names (zone = true)

Caches are populated at startup by LNS_DB.LoadRuleDB() and updated whenever rules change (via addRule(), setGlobalItem(), setPersonalItem(), or BulkSet()). The database is always written first, then the cache is updated to match.

Note: The upsert pattern (INSERT ... ON CONFLICT(item_id) DO UPDATE SET ...) is used throughout the database layer. This means you never need to check whether a row exists before writing—the database handles it atomically. If the item_id already exists, the row is updated; if not, a new row is inserted.
Warning: Items that have rules but are not yet in Items.db are assigned negative IDs. These are temporary placeholders generated by GetLowestID(), which finds the current minimum ID in the table and subtracts 1. When the actual item is later encountered in-game, the rule can be resolved to the real positive ID. Negative-ID items are tracked in LNS.GlobalItemsMissing and LNS.GlobalMissingNames.
