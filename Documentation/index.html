LootNScoot Documentation

This documentation targets new developers who may have zero prior experience with MacroQuest, EverQuest scripting, or the Lua programming language. Every concept is explained from the ground up so you can confidently read and modify the LootNScoot source code.

What is LootNScoot?

LootNScoot is an automation script for the MMORPG EverQuest (EQ). It handles the tedious task of looting items from defeated monsters, deciding what to keep, sell, bank, tribute, or destroy, and coordinating those decisions across multiple characters, even across separate PCs on the same network.

Before diving in, here are three foundational terms you will see everywhere:

Term What It Is
MacroQuest (MQ) A third-party tool that injects into the EverQuest client and exposes a rich scripting API. MQ provides Top Level Objects (TLOs — read-only accessors like mq.TLO.Me.Name()) to read game state (your character's name, nearby spawns (entities in the game world like NPCs, players, and corpses), inventory contents) and mq.cmd / mq.cmdf to execute in-game slash commands (e.g., mq.cmd('/sit') makes your character sit down). MQ also bundles an embedded Lua runtime so scripts can be written in Lua instead of the older macro language.
Lua A lightweight, dynamically-typed scripting language. MQ embeds Lua 5.1 (via LuaJIT). All LootNScoot code is pure Lua — there is no compilation step, no build system. You edit .lua files and run them directly inside MQ.
ImGui An immediate-mode graphical user interface library. Instead of creating persistent widget objects, you call drawing functions every frame. MQ exposes ImGui through mq.imgui, and LootNScoot uses it to render its settings window, item tables, loot history, and rule editors.

LootNScoot makes decisions using a rule-based system. Each item in the game can be assigned an action (Keep, Sell, Bank, Tribute, Quest, Destroy, CanUse, Ask, or Ignore). Rules live in an SQLite database and are split into Global rules (shared across characters on a server) and Personal rules (per-character overrides). Personal rules always take priority over Global rules, allowing each character to customize loot behavior without affecting others. When the script encounters an item it has never seen, it applies auto-rules based on your settings or prompts you to decide.

Source File Structure

The project lives in a single folder (lootnscoot/) inside MacroQuest's lua directory. Here is every source file and its role:

File ~Lines Purpose
init.lua 4,488 Core engine. Entry point, main loop, rule evaluation (getRule, lookupLootRule), looting logic (lootMobs, lootCorpse), item processing (sell, bank, tribute, destroy), settings I/O, CLI binds, event handlers.
modules/actor.lua 595 Multi-character communication via MQ's actor system. Sends and receives messages for loot coordination, settings sync, master looter commands, and directed-mode control.
modules/db.lua ~1,240 Database layer. Opens/creates SQLite databases (Items.db, AdvLootRules.db, LootHistory.db), provides prepared statements for CRUD operations, bulk set, import/export, wildcard rules, and historical data queries.
modules/loot_hist.lua ~1,000 Loot history tracking and the console/report UI. Records what was looted, by whom, and when. Provides the in-game loot console window and the report table.
modules/performance.lua 136 Lightweight performance profiling utilities for timing code sections during development and debugging.
modules/settings.lua 219 Declares every user-facing setting with its default value, tooltip text, and the TempSettings table used for inter-module communication between the UI and the main loop.
modules/ui.lua ~3,910 The entire ImGui user interface: main window, settings panel, rule tables (Global and Personal), item lookup, new-item popup, modify-item dialog, wildcard editor, buy-items list, historical data viewer, master looter window, and the toggle button.
lib/Logger.lua ~170 Logging utility with level filtering (debug, info, warn, trace, error, fatal, help). Outputs to both the MQ chat window and the in-game loot console.
lib/scrolling_plot_buffer.lua ~46 A ring-buffer data structure used by ImPlot to display scrolling performance graphs.
lib/themes.lua ~3,530 Comprehensive ImGui color theme definitions for consistent styling across the UI.

Key Dependencies

LootNScoot relies on several libraries and plugins. These are not bundled with the script — they come from MacroQuest or are fetched at runtime.

Dependency How It Is Loaded What It Provides
mq require 'mq' Core MacroQuest API. Provides mq.TLO for game data, mq.cmd/mq.cmdf for commands, mq.imgui for GUI, mq.pickle for serialization, mq.configDir for file paths, and mq.delay for yielding.
lsqlite3 PackageMan.Require('lsqlite3') SQLite3 bindings for Lua. Fetched automatically by MQ's PackageMan on first run. Used for all three databases (Items, Rules, History).
actors require('actors') MQ's built-in actor system for asynchronous inter-process messaging — it lets separate EQ instances (each running their own copy of LootNScoot) exchange data without sharing memory. Loaded by modules/actor.lua which wraps it. Characters send and receive Lua tables as messages to coordinate looting, share settings, and manage master looter state. See Multi-Character for details.
ImGui Built into MQ (mq.imgui) Immediate-mode GUI rendering. All windows, tables, buttons, checkboxes, and input fields are drawn through ImGui each frame.
ImPlot Built into MQ Plotting extension for ImGui. Used for scrolling performance graphs in the debug/performance view.
mq.PackageMan require('mq.PackageMan') MQ's built-in package manager. Automatically downloads and caches Lua packages (like lsqlite3) the first time they are required.
mq.Utils require('mq.Utils') File system utilities provided by MQ. Used for checking file existence, directory creation, and path manipulation.
MQ2Nav plugin /plugin Nav in MQ MQ2Nav is a navigation mesh plugin that provides pathfinding — it calculates a walkable route between two points in the game world, avoiding walls and obstacles. LootNScoot uses it to walk your character to corpses automatically. The script will refuse to start if MQ2Nav is not loaded.

How to Run

LootNScoot supports three operating modes. You choose a mode by passing command-line arguments when you start the script from the MQ in-game command line.

Once Mode (default)

/lua run lootnscoot

Performs a single action and then exits. This is the recommended way to run the script for the very first time — on a single character — so that PackageMan can install dependencies and all databases and config directories are created without race conditions. After the initial setup, you can launch all characters simultaneously in standalone mode.

You can also pass extra arguments to perform a specific action and exit: 

/lua run lootnscoot sellstuff
/lua run lootnscoot bank
/lua run lootnscoot tributestuff

Standalone Mode

/lua run lootnscoot standalone

Runs continuously in the background. The main loop checks for nearby corpses on a timer (controlled by LootCheckDelay), processes sell/bank/tribute queues, and keeps the UI alive. This is the most common mode for everyday use.

Directed Mode

/lua run lootnscoot directed scriptname

Designed to be launched by another script (such as RGMercs). In this mode, LootNScoot only loots when it receives a command over the actor system or when you manually issue /lns loot. The script will terminate automatically if the directing script stops running.

If you embed LootNScoot inside another script's folder, the path would be something like /lua run yourscript/lib/lootnscoot directed scriptname.
