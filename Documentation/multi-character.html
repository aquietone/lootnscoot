Multi-Character Features

Overview

EverQuest players commonly run multiple characters ("boxing") on separate game clients simultaneously. MacroQuest (MQ) provides an Actor system that allows Lua scripts running in different MQ instances to send messages to each other. The actor system works across the network between separate PCs and machines, not just between EQ instances on the same computer.

Cross-machine setup: For actor communication between MQ instances running on different PCs, network configuration may be required. You may need to add the IP addresses of your other machines to the MQ configuration so that actor messages can be routed across the network. Refer to your MacroQuest documentation for details on configuring cross-machine actor connectivity.

LootNScoot uses the Actor system to coordinate looting across all your running characters. When one character discovers a new item, sets a rule, changes a setting, or enters Master Looter mode, that information is broadcast to every other instance of LootNScoot. The Actor module lives in modules/actor.lua.

Actor Registration

When LootNScoot initializes, it calls LNS_ACTORS.RegisterActors(), which creates a mailbox named lootnscoot: 

function LNS_ACTORS.RegisterActors()
    LNS_ACTORS.lootActor = Actors.register('lootnscoot', function(message)
        local success, result = pcall(callback, message)
        if not success then
            Logger.Info(guiLoot.console,
                string.format('Actor callback failed: %s', result))
        end
    end)
end

Actors.register() comes from the actors MQ Lua library. It registers a named mailbox and provides a callback function that is invoked whenever a message arrives. The callback is wrapped in pcall() — Lua's protected call, which runs a function inside an error handler. If the function throws an error, pcall() catches it and returns false plus the error message, instead of crashing. This is critical here because actor messages come from other processes: a malformed or unexpected message should never take down the entire script. The error is logged and execution continues normally.

After registration, the script sends a Hello message to announce its presence: 

actors.Send({ action = 'Hello', Server = settings.EqServer, who = settings.MyName, })

Message Format

Actor messages are plain Lua tables. When you call LNS_ACTORS.Send(message), the table is serialized, transmitted to other MQ instances, and deserialized on the receiving end. The callback receives a callable object; calling message() extracts the Lua table.

Common fields found in LootNScoot actor messages:

FieldTypeDescription
whostringCharacter name of the sender.
actionstringThe action identifier (see Action Reference below).
ServerstringEverQuest server name (spaces replaced with underscores). Used for filtering.
itemIDnumberDatabase ID of the item.
itemstringItem name (the item_name or item field).
rulestringThe loot rule (Keep, Sell, Ignore, etc.).
sectionstringWhich rules table: GlobalItems or PersonalItems.
classesstringClass restriction string (e.g., "shm clr dru" or "All").
linkstringEverQuest item link string for in-game inspection.
settingstableFull settings table (used in settings sync messages).
CorpseIDnumberSpawn ID of the corpse (used in Master Looter messages).
CountnumberHow many of an item the sender has (inventory + bank).
Warning: Every message includes a Server field. When a message arrives, the callback checks whether Server matches the local settings.EqServer. If it does not match (and is not 'NULL'), the message is silently discarded. This prevents cross-server interference when you have characters on different servers running simultaneously.

Action Reference

The action field in each message determines what the receiver does with it. Here is a complete reference of every action string the actor callback handles:

ActionDirectionDescription
Hello Broadcast Character announces it is online. Triggers the receiver to send its settings back and add the sender to LNS.BoxKeys (the list of known characters).
sendsettings Broadcast Sends this character's full settings table to others. Sent via SendMySettings() after receiving a Hello, on init, and after setting changes.
updatesettings Broadcast Pushes updated settings to all characters. If the receiver's own name matches who, it applies the settings locally (used for remote setting changes from another character's UI).
addrule Broadcast A new loot rule was added for an item. Global rules are applied by all receivers; personal rules only by the matching character.
modifyitem Broadcast An existing item rule was changed. Same handling as addrule.
deleteitem Broadcast An item rule was deleted. The receiver removes it from their in-memory rule tables.
new Broadcast A new (previously unseen) item was found on a corpse. Other characters add it to their LNS.NewItems table and increment NewItemsCount. If AutoShowNewItem is enabled, the New Item popup opens.
check_item Broadcast Master Looter asks all characters to report how many of an item they own. Each receiver counts inventory + bank via mq.TLO.FindItemCount() and mq.TLO.FindItemBankCount(), then sends a response.
recheck_item Broadcast Re-requests item counts from all characters (used after the initial check to refresh stale data).
loot_item Directed (to specific character) Master Looter assigns an item to a specific character. The receiver adds the item to settings.TempSettings.ItemsToLoot, and the main loop processes it via LootItemML().
item_gone Broadcast An item was looted from a corpse and is no longer available. Receivers remove it from their Master Loot list.
corpse_gone Broadcast A corpse has been fully looted or has despawned. Receivers remove the entire corpse entry from the Master Loot list.
master_looter Broadcast Toggles Master Looter mode on or off. Receivers update their MasterLooting setting to match.
share_personal_rule Broadcast Shares a personal item rule to other characters. Each receiver calls setPersonalItem() to save it in their own personal rules table.
share_buy_item Broadcast Shares a buy-item entry (for vendor restocking) to other characters. Receivers call setBuyItem().
addsafezone Broadcast Adds a zone to the safe zone list (zones where looting is disabled). Receivers update LNS.SafeZones.
removesafezone Broadcast Removes a zone from the safe zone list.
updatewildcard Broadcast A wildcard rule was changed. Receivers set NeedReloadWildCards flag so the main loop reloads wildcard patterns.
reloadrules Broadcast Sent after a BulkSet operation. Contains the full updated rules, classes, and links tables. Receivers replace their in-memory rule tables entirely.
ItemsDB_UPDATE Broadcast Notifies other characters that the items database has been updated.

Directed Mode

In normal standalone operation, LootNScoot decides when to loot on its own. In Directed Mode, another Lua script takes control and tells LootNScoot when to loot. This mode exists because external scripts like RGMercs need precise control over looting timing — for example, only looting between combat rounds, not while the healer is casting, or only after all mobs in a pull are dead. Without directed mode, LootNScoot's automatic looting could interrupt the controlling script's workflow at the wrong moment.

How It Works

To run LootNScoot in directed mode: 

/lua run lootnscoot directed scriptname

This sets LNS.Mode = 'directed' and stores the directing script's name in LNS.DirectorScript. In directed mode, LootNScoot does not loot automatically. Instead, it waits for a doloot message from the director script. When looting finishes, LootNScoot sends a FinishedLooting() response back.

The director script can also query and modify LootNScoot's settings remotely, and check combat looting status. If the director script stops running (detected by checking mq.TLO.Lua.Script(DirectorScript).Status()), LootNScoot terminates itself.

Directed Actions

Directed mode uses the directions field (instead of action) to distinguish its messages from normal actor traffic:

DirectionSenderDescription
doloot Director → LNS Tells LootNScoot to start a loot cycle. Sets LNS.LootNow = true. May include a limit field for max corpses.
setsetting_directed Director → LNS Updates settings: CombatLooting, CorpseRadius, LootMyCorpse, IgnoreMyNearCorpses.
combatlooting Director → LNS Same as setsetting_directed (handled by the same code path).
getsettings_directed Director → LNS Requests LNS to send its current settings back to the director via the loot_module mailbox.
getcombatsetting Director → LNS Same as getsettings_directed (handled by the same code path).

LootNScoot's responses back to the director include:

SubjectSenderDescription
done_looting LNS → Director Sent by FinishedLooting() when the loot cycle completes.
processing LNS → Director Sent by InformProcessing() when LNS begins processing items (selling, tributing, etc.).
done_processing LNS → Director Sent by DoneProcessing() when item processing finishes.
mysetting LNS → Director Response to getsettings_directed, contains full settings and looted corpses list.
settings LNS → Director Sent on init and zone changes, includes combat/corpse settings.

Mailbox Targets

The Send() function routes messages to different mailboxes depending on the target: 

function LNS_ACTORS.Send(message, mailbox)
    message.Server = settings.EqServer
    if mailbox == 'loot_module' then
        LNS_ACTORS.lootActor:send(
            { mailbox = mailbox, script = LNS.DirectorScript, }, message)
    elseif mailbox == 'looted' then
        LNS_ACTORS.lootActor:send(
            { mailbox = mailbox, script = LNS.DirectorLNSPath or 'lootnscoot', }, message)
    else
        LNS_ACTORS.lootActor:send(
            { mailbox = 'lootnscoot', script = LNS.DirectorLNSPath or 'lootnscoot', }, message)
    end
end
Mailbox ParamTarget MailboxTarget ScriptPurpose
nil (default) lootnscoot LNS.DirectorLNSPath or 'lootnscoot' Normal broadcast to all LootNScoot instances.
'loot_module' loot_module LNS.DirectorScript Directed-mode responses back to the controlling script.
'looted' looted LNS.DirectorLNSPath or 'lootnscoot' Loot event notifications to the embedded version of the script.

Master Looter Mode

Master Looter mode allows one character to coordinate item distribution across the group. Instead of each character independently deciding what to loot, the "master" sees all items on corpses and manually assigns who gets what.

Workflow

  1. Item Discovery: When a character finds an item on a corpse, InsertMasterLootList() adds it to the local MasterLootList data structure and broadcasts a check_item message. This message includes the item name, link, corpse ID, item value, NoDrop status, and the sender's own item count.
  2. Count Response: Other characters receive check_item, count how many of that item they own (using mq.TLO.FindItemCount() + mq.TLO.FindItemBankCount()), and send their count back as another check_item message. They also mark the corpse ID as looted to avoid re-looting.
  3. UI Display: The Master Looter sees the Master Looter Window (described on the User Interface page) with all corpses, their items, and each character's count for each item.
  4. Assignment: The Master clicks "Loot for [CharName]", which sends a loot_item action to that specific character with the corpse ID and item name.
  5. Pickup: The target character receives loot_item, adds the item to settings.TempSettings.ItemsToLoot, and the main loop processes it by navigating to the corpse and looting the specific item via LootItemML().
  6. Cleanup: When items are looted or corpses despawn, item_gone and corpse_gone messages remove entries from everyone's Master Loot list.

MasterLootList Structure

The LNS.MasterLootList is a nested Lua table with this structure: 

MasterLootList[corpseID] = {
    Items = {
        ["Item Name"] = {
            Link    = "item link string",
            Value   = 1234,
            NoDrop  = false,
            Lore    = false,
            Members = {
                ["CharName1"] = 3,    -- has 3 of this item
                ["CharName2"] = 0,    -- has 0
                ["CharName3"] = 1,    -- has 1
            },
        },
        ["Another Item"] = { ... },
    },
}

Each corpse is keyed by its spawn ID. Each item on that corpse tracks its link, value, flags, and a Members subtable mapping character names to their inventory counts.

Settings Synchronization

When a character starts up and sends a Hello message, other characters respond by setting the SendSettings flag, which triggers SendMySettings() in the main loop. This function packages the full settings.Settings table into a sendsettings message: 

function LNS_ACTORS.SendMySettings()
    LNS.Boxes[settings.MyName] = settings.Settings
    LNS_ACTORS.Send({
        who             = settings.MyName,
        action          = 'sendsettings',
        settings        = settings.Settings,
        CorpsesToIgnore = LNS.lootedCorpses or {},
        CombatLooting   = settings.Settings.CombatLooting,
        CorpseRadius    = settings.Settings.CorpseRadius,
        LootMyCorpse    = settings.Settings.LootMyCorpse,
        IgnoreNearby    = settings.Settings.IgnoreMyNearCorpses,
        Server          = settings.EqServer,
    })
end

Received settings are stored in LNS.Boxes[who] (a table of character-name to settings). The Settings UI can then display a dropdown of all known characters and let you view or modify any character's settings. When you modify another character's settings through the UI, an updatesettings message is sent, and the target character applies the changes locally.

Tip: You can view and change any character's settings remotely from any other character's Settings UI. Select the target character from the Actor Selector dropdown. Changes are sent via the updatesettings actor message and applied immediately on the target.

Debug Mailbox

The function DebugMailBox() renders an overlay window that displays all actor messages in real time. It is enabled by:

  • Running /lns debug on which sets LNS.debugPrint = true.
  • Running /lns mailbox which toggles settings.TempSettings.ShowMailbox.

When debug mode is active, every incoming message is recorded in settings.TempSettings.MailBox as a table with Time, Subject (the action or direction string), and Sender fields. The mailbox viewer displays these entries sorted by time (newest first), giving you a live view of all actor traffic.

Note: The actor system tracks Messages Per Second (MPS). The callback measures time between messages and maintains a running average in settings.TempSettings.MPS. After 10 seconds of inactivity, the counter resets. This metric is useful for diagnosing performance issues when many characters are broadcasting simultaneously.
