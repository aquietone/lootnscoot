Architecture

This page explains how LootNScoot is structured internally: how the files relate to each other, how data flows between modules, and what the main loop does on every tick. Understanding this architecture is essential before modifying or extending the script.

High-Level Overview

LootNScoot follows a hub-and-spoke architecture:

  • Hub — init.lua is the central file. It contains the entry point, the main loop, the rule evaluation engine, the looting logic, and the settings I/O. It creates and owns the LNS table (the shared state object) and passes it to every module.
  • Spokes (modules/) — Each module in the modules/ directory handles one domain: actor.lua for messaging, db.lua for databases, ui.lua for the GUI, settings.lua for configuration declarations, loot_hist.lua for history, and performance.lua for profiling.
  • Utilities (lib/) — Small helper libraries in lib/ that are used by the modules: logging, plot buffers, and themes. These have no knowledge of LootNScoot's domain logic.

Notice that init.lua requires all the modules at the top of the file, but the modules do not require init.lua back. Instead, they receive the shared LNS state table through the SetLNS() pattern described below.

The LNS Table

LNS is a plain Lua table declared at the top of init.lua: 

local LNS = {}

This single table is the central state object for the entire script. Almost everything that needs to be shared between modules is stored here. Here are the most important fields grouped by category:

Rule Data

These tables are in-memory caches of what is stored in the SQLite database. They exist for performance — during looting, every item on every corpse needs a rule lookup, and querying the database each time would be too slow. Instead, rules are cached in plain Lua tables for instant key-based access.

FieldTypeDescription
GlobalItemsRules table {[itemID] = ruleString} In-memory cache of Global loot rules, keyed by item ID
GlobalItemsClasses table {[itemID] = classString} Class restrictions for global rules (e.g., "WAR CLR" or "All")
PersonalItemsRules table {[itemID] = ruleString} In-memory cache of Personal (per-character) rules
PersonalItemsClasses table {[itemID] = classString} Class restrictions for personal rules
WildCards table (array of {wildcard, rule}) Pattern-based rules that match item names using glob syntax
ItemLinks table {[itemID] = linkString} Cached item links for right-click inspection in the UI
ItemNames table {[itemID] = nameString} Cached item names for display in tables

The SetLNS Pattern

Since Lua's require() system caches modules on first load, you cannot have circular dependencies. The modules need access to the LNS table defined in init.lua, but they are loaded before LNS is populated. The solution is a one-way injection pattern: 

-- In init.lua, after LNS is created and populated:
db.SetLNS(LNS)
actors.SetLNS(LNS)
ui.SetLNS(LNS)

-- Inside each module (e.g., modules/db.lua):
local LNS = {}

function module.SetLNS(_LNS)
    LNS = _LNS
end

Each module declares a local LNS variable (initially empty) and exposes a SetLNS() function. During initialization, init.lua calls each module's SetLNS() and passes in the real LNS table. From that point on, the module's local LNS variable points to the same table in memory, so any changes made by any module are visible to all others.

This works because Lua tables are reference types. When you assign LNS = _LNS, both variables point to the same table object. There is no copying involved.

Main Loop

LNS.MainLoop() is a while not LNS.Terminate do loop that runs continuously. Each iteration (called a "tick") performs the following checks and actions in order. The loop ends with a mq.delay(10) call — an MQ function that pauses script execution for approximately 10 milliseconds, yielding control back to MQ so it can process other tasks (rendering, other scripts, game events) before the next tick.

MainLoop tick: +----------------------------------------------------------+ | 1. Update Zone and Instance | | LNS.Zone = mq.TLO.Zone.ShortName() | | LNS.Instance = mq.TLO.Me.Instance() | | | | 2. Game State Check | | if GameState != "INGAME" --> Terminate | | | | 3. Director Check (directed mode only) | | if director script stopped --> Terminate | | | | 4. Zone Change Detection | | if zone changed: clear corpse list, reset state, | | notify director, clear master loot list | | | | 5. Safe Zone Check | | if in safe zone: warn once, skip looting | | | | 6. Process TempSettings Flags (pre-loot) | | DoImport, RemoveSafeZone, RemoveMLItem, | | RemoveMLCorpse, ItemsToLoot, SendLoot, | | SendRemoveItem, AddWildCard, DeleteWildCard, | | NeedReloadWildCards | | | | 7. Loot Mobs (if DoLoot and not directed) | | or Loot Now (if directed and LootNow == true) | | Respects LootCheckDelay timing | | | | 8. Process Items State Machine | | doSell --> processItems('Sell') | | doBuy --> processItems('Buy') | | doTribute --> processItems('Tribute') | | doBulkSet --> db.BulkSet() | | | | 9. History Updates | | Sync guiLoot report and console visibility | | Load historical data if requested | | | | 10. Process Events | | mq.doevents() | | | | 11. Settings Save (if NeedSave flag is set) | | Update BuyItems, pickle settings, reload, | | notify actors, re-sort tables | | | | 13. Item Lookups | | Process GetItems queue, search requests | | | | 14. New Items Counter | | Reset showNewItem if count reaches zero | | | | 15. Cleanup (if NeedsCleanup and not casting) | | processItems('Destroy') | | | | 16. mq.delay(10) -- yield to MQ | +----------------------------------------------------------+

This order matters. Zone changes are detected early so the rest of the tick operates on the correct zone data. TempSettings flags are processed before looting so that UI actions (like adding a wildcard) take effect immediately. Settings saves happen near the end so that all changes from the current tick are captured. For details on the looting steps (7), see the Looting Engine page. For the rule evaluation used during looting, see the Rule System.

The TempSettings Pattern

One of the most important architectural patterns in LootNScoot is the TempSettings table. This is the primary mechanism for communication between the UI module and the main loop.

The challenge: ImGui (the immediate-mode GUI library — see the Home page) rendering happens on a separate callback (ui.RenderUIs() is called by MQ every frame), while the main loop runs in LNS.MainLoop(). These two cannot directly call each other's functions safely — if a button click triggered a slow database import inside the render callback, the entire UI would freeze. Instead, they communicate through shared boolean flags in settings.TempSettings, and the main loop performs the actual work on the next tick.

How It Works

  1. UI sets a flag — When the user clicks "Save" in the UI, the render function sets settings.TempSettings.NeedSave = true.
  2. Main loop reads the flag — On the next tick, the main loop checks if settings.TempSettings.NeedSave then and performs the actual save operation.
  3. Main loop clears the flag — After completing the operation, the main loop sets settings.TempSettings.NeedSave = false.

Here are the most commonly used TempSettings flags:

FlagSet ByRead ByAction
NeedSave UI, actors MainLoop Write settings to disk, reload, notify actors
UpdateSettings UI MainLoop Pickle settings and reload them
SendSettings UI MainLoop Broadcast settings to other characters via actors
DoImport UI MainLoop Import an old rules database file
AddWildCard UI MainLoop Add a new wildcard pattern rule
DeleteWildCard UI MainLoop Remove a wildcard pattern rule
NeedReloadWildCards UI, db MainLoop Reload wildcard rules from the database into memory
doBulkSet UI MainLoop Apply a rule to multiple items at once
NeedsCleanup UI, binds MainLoop Destroy items marked for destruction
LookUpItem UI MainLoop Search the Items.db for an item by name
LookUpDateData UI MainLoop Load historical loot data for a specific date
Why not call functions directly? In an immediate-mode GUI, the render function is called every frame (potentially 60+ times per second) by the host application (MQ). If the render function triggered a lengthy operation like a database import, it would block rendering and freeze the UI. By setting a flag and letting the main loop handle it, the work happens at the appropriate time in the script's execution cycle, and the UI remains responsive.

The TempSettings table also carries data payloads alongside flags. For example, when adding a wildcard, the UI sets three values: 

settings.TempSettings.AddWildCard = true
settings.TempSettings.NewWildCard = "Spell:*"
settings.TempSettings.NewWildCardRule = "Keep"

The main loop reads all three, performs the database operation, and then clears them: 

if settings.TempSettings.AddWildCard then
    if settings.TempSettings.NewWildCard ~= '' then
        db.AddWildCard(settings.TempSettings.NewWildCard,
                       settings.TempSettings.NewWildCardRule)
        settings.TempSettings.NewWildCard = ''
        settings.TempSettings.NewWildCardRule = 'Ask'
    end
    settings.TempSettings.AddWildCard = false
end

This pattern appears dozens of times throughout the main loop. Recognizing it is the key to understanding how user actions in the GUI translate into actual operations in the script.
