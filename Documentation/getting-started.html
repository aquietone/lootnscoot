Getting Started

This page walks you through everything you need before running LootNScoot for the first time, explains what happens during startup, and describes your first looting experience step by step.

Prerequisites

LootNScoot needs a few things in place before it can run. If any of these are missing, the script will either refuse to start or will lack critical functionality.

MQ2Nav Plugin (Required)

MQ2Nav is a MacroQuest (MQ) plugin that provides navigation mesh pathfinding — it calculates walkable routes through the 3D game world, letting scripts move your character automatically. LootNScoot uses it to walk your character to corpses. The very first thing the script does on startup is check for MQ2Nav using a TLO (Top Level Object — MQ's read-only API for accessing game data): 

if not mq.TLO.Plugin("MQ2Nav").IsLoaded() then
    print("WARNING!! MQ2Nav is NOT loaded...")
    mq.exit()
end

If MQ2Nav is not loaded, the script prints a warning and exits immediately.

MQ2Nav is mandatory. Load it before starting LootNScoot by typing /plugin Nav in the MQ command line. On most setups, you can add this to your MQ auto-load configuration so it starts with every session.

lsqlite3 (Auto-installed)

LootNScoot stores all its data in SQLite databases. It uses the lsqlite3 Lua binding to talk to SQLite. This library is fetched automatically by MQ's PackageMan system the first time you run the script: 

local PackageMan = require('mq.PackageMan')
local sql = PackageMan.Require('lsqlite3')

PackageMan is MQ's built-in package manager. When you call PackageMan.Require('lsqlite3'), it checks if the package is already cached locally. If not, it downloads and installs it. This happens silently on the first run — you do not need to install anything manually.

Items.db (Optional but Recommended)

The Items database contains metadata for every EverQuest item: name, sell value, tribute value, class restrictions, whether it is NoDrop (cannot be traded to other players), Lore (only one can be held at a time), tradeskill, and so on. LootNScoot will create an empty database if none exists and populate it as you encounter items, but starting with a pre-built database gives you immediate access to over 117,000 items.

A default Items.db covering all standard EQEmu server items is available from:
https://github.com/grimmier378/emu_itemdb/releases

Place the downloaded Items.db file at:
[MQ Dir]/resources/LootNScoot/[ServerName]/Items.db

If your server has customized items, the database will automatically update with correct data as you loot or import those items. The pre-built database is just a head start.

Installation

Installation is simple — there is no build step, no compilation, and no package to install beyond placing files in the right directory.

  1. Locate your MacroQuest lua directory. This is typically [MQ Install Dir]/lua/.
  2. Place the entire lootnscoot/ folder inside the lua directory so the entry point is at [MQ Dir]/lua/lootnscoot/init.lua.
  3. Ensure MQ2Nav is loaded (/plugin Nav).
  4. Run the script once on a single character: /lua run lootnscoot (no arguments).

Running without arguments performs the initial setup — PackageMan downloads lsqlite3, the script creates its config and database directories, and then it exits cleanly. This one-time setup should be done on a single character only to avoid race conditions (multiple characters trying to create the same directories and databases simultaneously).

First launch: one character only. Run /lua run lootnscoot on a single character and let it finish before starting the script on other characters. Once the initial setup completes, you can safely load all characters at once with /lua run lootnscoot standalone.
After your first run, consider these common setup steps: (1) Run /lns importinv to seed the Items database with your existing gear. (2) Open the UI and review the Settings tab. (3) Kill a few enemies and check the New Items popup to verify auto-rules match your preferences.

First Run

When you type /lua run lootnscoot for the first time (no arguments), the script executes init.lua from top to bottom, performs all setup, and then exits. Here is exactly what happens inside the LNS.init(args) function, in order:

  1. Database module initialized — db.SetLNS(LNS) gives the database module a reference to the central state table so it can read and write shared data.
  2. Settings loaded or created — LNS.loadSettings(true) reads your character's settings file. If no file exists (first run), it creates one with all default values from modules/settings.lua and returns true (needs save).
  3. Prepared statements created — db.PrepareStatements() compiles all SQL queries into reusable prepared statements for performance.
  4. Map filter updated — /squelch /mapfilter CastRadius is set to the configured CorpseRadius value.
  5. Settings sorted — LNS.SortSettings() and LNS.SortTables() organize settings keys alphabetically for the UI.
  6. Actors registered — actors.SetLNS(LNS) and actors.RegisterActors() set up the inter-process messaging system.
  7. Bag space checked — LNS.CheckBags() counts your available inventory slots.
  8. Events bound — LNS.setupEvents() registers MQ event handlers for detecting sells, tributes, and other game events that LootNScoot should react to.
  9. CLI commands registered — LNS.setupBinds() creates the /lns and /lootutils slash commands so you can control the script from the MQ command line. See CLI Commands for the full list.
  10. Arguments parsed — processArgs(args) reads the command-line arguments (standalone, directed scriptname, etc.) to determine the operating mode. The mode is stored in LNS.Mode. A "Hello" message is broadcast to other running instances if running in standalone mode.
  11. Settings broadcast — actors.SendMySettings() sends this character's settings to all other running LootNScoot instances.
  12. ImGui initialized — mq.imgui.init('LootnScoot', ui.RenderUIs) registers the UI render function. From this point forward, MQ calls ui.RenderUIs() every frame to draw the LootNScoot windows.
  13. Settings saved if needed — If settings were newly created, they are written to disk via mq.pickle().
  14. Main loop entered — LNS.MainLoop() begins and runs until termination. See Architecture for details on the loop.

File Locations

LootNScoot organizes its files by server name and character name so you can play on multiple servers or with multiple characters without data conflicts.

[MQ Directory] | +-- lua/ | +-- lootnscoot/ <-- Script source code | +-- init.lua | +-- modules/ | | +-- actor.lua | | +-- db.lua | | +-- loot_hist.lua | | +-- performance.lua | | +-- settings.lua | | +-- ui.lua | +-- lib/ | +-- Logger.lua | +-- scrolling_plot_buffer.lua | +-- themes.lua | +-- config/ | +-- LootNScoot/ | +-- [ServerName]/ | +-- [CharName].lua <-- Per-character settings | +-- DB_Updated.lua <-- DB update tracking | +-- resources/ +-- LootNScoot/ +-- [ServerName]/ +-- Items.db <-- Item metadata database +-- AdvLootRules.db <-- Global + Personal loot rules +-- LootHistory.db <-- Historical loot records

The [ServerName] is the EQ server name with spaces replaced by underscores (e.g., Project_Lazarus). The [CharName] is your character's display name exactly as it appears in game.

The settings file uses the .lua extension because it is serialized with mq.pickle() — an MQ function that converts Lua tables to/from files. The output is valid Lua source code, which means the file is both human-readable (you can open it in a text editor) and directly loadable by Lua's dofile().

Your First Loot

Here is a narrative walkthrough of what happens the first time a corpse appears near your character while LootNScoot is running in standalone mode.

  1. Corpse detection — The main loop calls LNS.lootMobs(), which uses mq.TLO.SpawnCount('npccorpse radius 100 zradius 50') to find NPC corpses within your configured CorpseRadius (default 100 units) and CorpseZRadius (default 50 units, the vertical range).
  2. Safety checks — Before looting, the script checks: Are you invisible? Are hostile mobs too close (within MobsTooClose range, default 40)? Is your own corpse nearby? Are you in a safe zone? Is looting paused? If any check fails, looting is skipped this cycle.
  3. Navigation — The script calls LNS.navToID(corpseID) which uses MQ2Nav to pathfind your character to the corpse. Navigation has a configurable timeout (NavTimeout, default 5 seconds).
  4. Loot window opened — Once at the corpse, the script right-clicks it to open the EQ loot window. It waits up to a few seconds for the window to appear.
  5. Item evaluation — For each item on the corpse, the script calls LNS.getRule(item, 'loot'). This function looks up existing rules, applies your settings, and returns a decision. See the Rule System page for the full decision tree.
  6. Item pickup — Based on the decision, the script either loots the item (Keep, Sell, Bank, Tribute, Quest) or skips it (Ask, Ignore, CanUse, Destroy). Items picked up are recorded in the loot history.
  7. New item handling — If the script encounters an item it has never seen before, it creates a new rule using auto-rule logic (if UseAutoRules is enabled) and adds the item to the "New Items" list. A gold coin icon appears on the toggle button to alert you.
  8. Loot window closed — After processing all items, the script clicks the Done button to close the loot window and moves on to the next corpse (up to MaxCorpsesPerCycle, default 5).
  9. Actor notification — Loot data is broadcast to other running LootNScoot instances via the actor messaging system so they can update their history and master looter state.

Common First Steps

Setting Up Rules

When you first start using LootNScoot, you will not have any rules in your database. The script handles this gracefully:

  • If UseAutoRules is true (the default), the script automatically assigns rules to new items based on their properties. Items with a sell value get Sell, items with tribute value get Tribute, equippable items get Keep, and No Drop equippable items get CanUse.
  • Items that cannot be auto-ruled are set to Ask, which means the script will skip them and add them to the New Items list for you to decide.
  • You can review and change new item rules through the UI (left-click the toggle button) or the New Items window (right-click the toggle button or /lns shownew).

Importing Your Inventory

If you want to pre-populate the Items database with everything your character already has, use: 

/lns importinv

This scans your inventory and bank (if bank data is available) and adds every item to the Items.db database. This is especially useful when starting fresh or migrating to a new server, as it gives the rule system knowledge of items you already own.

Bank data may not be current unless you have recently visited your bank or zoned. For the most complete import, visit your bank first and then run the import command.

Reviewing Settings

Open the main LootNScoot window (/lns show or left-click the toggle button) and navigate to the Settings tab. Every setting has a tooltip explaining what it does. Start with these key settings:

  • CorpseRadius (default 100) — How far to search for corpses
  • SaveBagSlots (default 3) — Stop looting when this many bag slots remain empty
  • MinSellPrice (default -1, meaning any) — Minimum plat value to auto-loot sell items
  • DoLoot (default true) — Master switch for automatic looting
  • LootNoDrop (default false) — Whether to pick up NoDrop items (items that cannot be traded once looted)

For a complete list of every setting, see the Settings Reference.
