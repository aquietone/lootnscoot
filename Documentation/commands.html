CLI Commands

Overview

LootNScoot registers two command binds that both call the same handler function, LNS.commandHandler(). mq.bind() is a MacroQuest (MQ) function that registers a slash command so that typing it in the MQ console (or from a macro) runs a Lua function: 

mq.bind('/lootutils', LNS.commandHandler)
mq.bind('/lns', LNS.commandHandler)

You can use either /lns or /lootutils interchangeably in the MQ console or from macros. All examples on this page use /lns for brevity.

Commands are parsed by splitting the input string into an args table. The first argument determines the command, and subsequent arguments provide parameters. Quoted strings are treated as a single argument (e.g., /lns sell "Rusty Sword" passes "Rusty Sword" as one argument).

Quick Action Commands

These commands trigger immediate actions. They are single-word commands with no additional arguments required.

CommandDescriptionExample
loot Trigger an immediate looting cycle. Scans for corpses within range and loots up to MaxCorpsesPerCycle corpses. See Looting Engine. /lns loot
sellstuff Sell all items in your inventory that have a "Sell" rule. Requires targeting a merchant NPC. Reports profit when done. /lns sellstuff
bankstuff Bank all items in your inventory that have a "Bank" rule. Requires targeting a Banker NPC. /lns bankstuff
tributestuff Tribute all items in your inventory that have a "Tribute" rule. Requires targeting a Tribute Master NPC. /lns tributestuff
cleanup Destroy all items in your inventory that have a "Destroy" rule. Requires the DoDestroy setting to be enabled. /lns cleanup
restock Buy items from the BuyItemsTable. Requires targeting a merchant who sells the items. /lns restock
pause Pause automatic looting. Sets LNS.PauseLooting = true. /lns pause
unpause / resume Resume automatic looting. Sets LNS.PauseLooting = false. Both words do the same thing. /lns unpause
corpsereset Clear the looted corpse tracking list (LNS.lootedCorpses). Useful when CheckCorpseOnce is on and you want to re-loot corpses you already visited. /lns corpsereset

Rule Commands

Rule commands let you set loot rules for items. There are several ways to specify which item you mean and whether the rule applies globally (all characters) or personally (this character only).

Valid rule actions: ask, keep, bank, sell, ignore, destroy, quest, tribute, canuse. These are case-insensitive on input but stored in proper case (e.g., "Keep", "Sell").

Quick Rule (Cursor Item)

The fastest way to set a rule: put an item on your cursor (by picking it up from inventory), then type the action as a single command. The rule is applied to the Global_Rules table.

CommandEffect
/lns keepMark cursor item as "Keep" globally
/lns sellMark cursor item as "Sell" globally
/lns bankMark cursor item as "Bank" globally
/lns ignoreMark cursor item as "Ignore" globally
/lns destroyMark cursor item as "Destroy" globally
/lns tributeMark cursor item as "Tribute" globally
/lns canuseMark cursor item as "CanUse" globally
/lns askMark cursor item as "Ask" globally
Tip: Putting an item on your cursor is the quickest way to set rules. Pick up the item, type /lns sell, and the rule is saved immediately using the item's actual game ID and item link.

Quest Format

For quest items where you want to keep a specific quantity, use the pipe syntax with the item on your cursor: 

/lns quest|5

This marks the cursor item as "Quest|5", meaning LootNScoot will loot it until you have 5 total (inventory + bank). The number after the pipe is the desired count. If no number is provided, the QuestKeep default (10) is used.

Rule by Name

You can set rules for items by name without having the item on cursor. The item must exist in the Items.db database: 

/lns sell "Rusty Sword"
/lns keep "Bone Chips"
/lns ignore "Bat Wing"

The script resolves the item name to an ID using resolveItemIDbyName(). If the item is found, the rule is applied to Global_Rules. If not found, it is logged as a warning.

Global vs Personal

By default, rules set via the quick methods above go into Global_Rules (shared across all characters on the server). To explicitly control which table a rule goes into, use the globalitem, normalitem, or personalitem prefixes:

CommandTableExample
globalitem Global_Rules /lns globalitem keep "Fire Opal"
normalitem Global_Rules (alias) /lns normalitem sell "Bat Wing"
personalitem [CharName]_Rules /lns personalitem ignore "Rusty Sword"

normalitem and globalitem are functionally identical—both write to Global_Rules. These commands also work with the item on cursor (omit the item name) or with a quoted item name.

Quest format works with these prefixes too: 

/lns globalitem quest "Bone Chips" 20
/lns personalitem quest "Fire Beetle Eye" 10

If the item is not found in Items.db, a rule is still created with a temporary negative ID (see Database Layer for details on negative IDs).

Configuration Commands

These commands let you view and change settings from the command line.

CommandDescriptionExample
set <name> <value> Change a setting. The name is case-insensitive (uses SettingsEnum for lookup). Boolean settings use on to enable (any other value, including off, disables). Number settings accept any numeric value. String settings accept any string. /lns set doloot on
/lns set corpseradius 150
/lns set lootchannel dgt
config Print all current setting names and values to the MQ console. Non-table, non-function settings are shown. /lns config
reload Reload settings from disk and refresh the UI. Useful after manually editing the settings file. /lns reload

Buy Item Commands

These commands manage the BuyItemsTable, which lists items to automatically purchase from vendors during restocking.

CommandDescriptionExample
buy (with cursor item) Add the item currently on your cursor to the buy list with quantity 1. Pick up item, then: /lns buy
buy <qty> (with cursor item) Add the cursor item with the specified quantity. Pick up item, then: /lns buy 20
buy "<name>" Add an item by name with quantity 1. /lns buy "Bone Chips"
buy "<name>" <qty> Add an item by name with the specified quantity. /lns buy "Bone Chips" 100

After adding a buy item, the settings are automatically saved to disk. To actually purchase the items, use /lns restock while targeting a merchant, or enable AutoRestock to buy automatically after selling.

Startup Arguments

LootNScoot accepts an argument when launched via /lua run (the MQ command that starts a Lua script). The argument determines the operating mode: 

/lua run lootnscoot [mode]
ArgumentDescription
standalone Run as an independent script with its own main loop. Loots automatically on a timer. This is the normal user-facing mode.
directed <script> [path] Run as a library controlled by another script (e.g., RGMercs). Looting only happens when the directing script sets LNS.LootNow = true. This mode exists because external automation frameworks need to control when looting occurs — for example, RGMercs only wants to loot between combat rounds, not during healing. See Multi-Character for details on directed-mode messaging.
once Perform a single looting cycle and then stop. Equivalent to calling lootMobs() once.
sellstuff Run processItems('Sell') and exit. Requires a merchant to be targeted.
bankstuff Run processItems('Bank') and exit. Requires a banker to be targeted.
tributestuff Run processItems('Tribute') and exit. Requires a tribute master to be targeted.
cleanup Set the NeedsCleanup flag for destroy-marked items to be processed.
restock Run processItems('Buy') and exit. Requires a merchant to be targeted.

If no argument is provided, the script runs in once mode — it performs initial setup (PackageMan installs, database and config directory creation) and then exits without entering the continuous main loop. This is the preferred first-run method on a single character to avoid race conditions during setup. After that, use standalone for persistent background looting.

TLO Access

LootNScoot registers a custom Top-Level Object (TLO) named LNS that other scripts and macros can query to check the script's status. A TLO is MQ's read-only data API — think of it as a variable that other scripts can read but not write. MQ has many built-in TLOs (like ${Me.Name} for your character name), and Lua scripts can register their own custom ones.

The TLO is registered via: 

mq.AddTopLevelObject('LNS', LNS.TLOHandler)

Available members:

TLO MemberReturn TypeDescription
${LNS.Looting} bool Returns true if LootNScoot is currently in a looting cycle (LNS.IsLooting).
${LNS.Paused} bool Returns true if looting is paused (LNS.PauseLooting).
${LNS.Mode} string Returns the current operating mode: "standalone", "directed", or "once".
${LNS.SafeZone} bool Returns true if the current zone is a safe zone. Optionally accepts a zone short name parameter to check a specific zone: ${LNS.SafeZone[zonename]}.

These TLO members can be used in MQ macros: 

/if (${LNS.Looting}) /echo LootNScoot is currently looting
/if (!${LNS.Paused}) /echo Looting is active

Or in other Lua scripts: 

local isLooting = mq.TLO.LNS.Looting()
local isPaused = mq.TLO.LNS.Paused()
local mode = mq.TLO.LNS.Mode()
